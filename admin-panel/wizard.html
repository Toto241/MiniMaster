Was wird hier eingerichtet?

MiniMaster braucht ein eigenes Firebase-Projekt. Wir sammeln in den naechsten Schritten drei Gruppen von Informationen:

  1. Firebase-Web-Konfiguration (API-Key, Project-ID, …) – damit die Browser-Panels (Admin, Web-Control, Eltern, Kind) sich beim Backend authentifizieren koennen.
  2. Pflicht-Dateien (zwei google-services.json plus serviceAccountKey.json) – fuer die Android-Apps und Server-seitige Admin-Operationen.
  3. Optionale Secrets (Gemini-/OpenAI-Key, App-Check Site-Key, Apple-Subscription-Felder) – fuer KI-Funktionen und externe Anbindungen.
Bevor du startest: Lege in der Firebase-Console einmalig ein Projekt an und registriere darin eine Web-App, eine Android-App com.minimaster.masterapp und eine Android-App com.google.pairing. Den Link findest du unten.

Firebase-Web-Konfiguration

Wofuer ist das wichtig? Diese Werte stehen in admin-panel/firebase-config.js (und in den drei anderen Panels). Sie sagen dem Browser, mit welchem Firebase-Projekt er sprechen soll. Ohne diese Werte kann sich kein Operator anmelden, kein Gerät pairen und keine Telemetrie geschrieben werden.
🌐 Direkt aus Firebase importieren (Google-Login, ohne CLI)

Was passiert hier? Du meldest dich mit deinem Google-Konto an, danach holt der Wizard die Web-Konfig und beide google-services.json-Dateien direkt von der Firebase Management REST API aus dem Internet. Es muss keine Firebase CLI installiert sein. Der Access Token bleibt nur im Browser-Speicher und wird nicht persistiert.

Einmal-Setup: OAuth-Client-ID anlegen (warum & wie)

Google erlaubt API-Zugriff nur ueber registrierte OAuth-Clients. Da MiniMaster nicht mit einer eigenen, geteilten Client-ID ausgeliefert wird (das wuerde alle Nutzer auf dieselbe Quote setzen und ist gegen die Google-Richtlinien), muss du eine eigene Client-ID anlegen – einmalig:

  1. Google Cloud Console → APIs & Services → Credentials oeffnen
  2. „Create Credentials" → „OAuth client ID"
  3. Application type: Web application
  4. Authorized JavaScript origins: http://127.0.0.1:8765 und http://localhost:8765
  5. Authorized redirect URIs: (leer lassen – wir nutzen den GIS-Token-Flow)
  6. „Create" → Client ID kopieren
  7. Zusaetzlich pruefen: Firebase Management API muss im Projekt aktiviert sein
  8. WICHTIG – sonst „Fehler 403: access_denied" beim Login: OAuth consent screen oeffnen → Status sollte „Testing" sein (Default) → unter „Test users" auf „+ Add users" → die eigene Google-Mail-Adresse eintragen. Solange der Consent-Screen unverifiziert ist, duerfen nur die hier gelisteten Konten den Login durchlaufen (Google-Standard fuer nicht-Workspace-Apps; bis zu 100 Tester moeglich).
    Falls Google meldet „not associated with a Google Account": Du hast vermutlich eine @googlemail.com-Adresse eingetragen. Google's Validator akzeptiert nur die kanonische @gmail.com-Form – beide gehoeren demselben Konto. Trage stattdessen name@gmail.com ein (Punkte in Gmail-Adressen werden ohnehin ignoriert).

Wird im Browser-Speicher (localStorage) gemerkt, sodass du sie bei der naechsten Sitzung nicht erneut eingeben musst.

Pflicht-Dateien aus der Firebase-Console

Wofuer ist das wichtig? Diese drei JSON-Dateien werden NICHT ueber die Web-Konfiguration abgedeckt und muessen separat aus der Firebase-Console heruntergeladen werden:
  • google-services.json der Android-Apps – binden die mobilen Apps an dein Projekt.
  • serviceAccountKey.json – erlaubt server-seitige Admin-Operationen (Setup-Admin-Befehle, Migrationen, Tests).
Ohne diese Dateien laufen die mobilen Apps nicht und die Pre-Flight-Pruefung bleibt rot.
🔧 Tool-Helfer: SHA-1 / SHA-256 aus Android-Keystore berechnen

Wofuer? Google Sign-In, Firebase App Check (Play Integrity) und Dynamic Links setzen voraus, dass der SHA-1-Fingerprint deines Signierungs-Zertifikats in der Firebase-Console hinterlegt ist – zusaetzlich zur google-services.json. Der Wert steht nicht in der heruntergeladenen JSON; er muss separat aus dem Keystore gelesen werden (Tool: keytool, Teil von JDK 17).

Workflow: Knopf druecken → SHA-1 + SHA-256 erscheinen unten → in der Firebase-Console unter Project Settings → Android-App → „Fingerprint hinzufuegen" einfuegen.

Wofuer ist das wichtig? Diese Werte sind nicht pflicht fuer den Start, schalten aber konkrete Features frei. Jedes Feld kann uebersprungen und spaeter im Admin-Panel nachgereicht werden.

🔗 Google AI Studio – Key erzeugen

Kurz: Wenn du nicht sicher bist, was das ist, lass das Feld leer – fuer lokale Entwicklung und Beta-Tests brauchst du es nicht.

Lang: Firebase App Check ist eine Bot-Bremse fuer dein Backend (Firestore/Functions/Storage). Es laesst nur Aufrufe von deinen echten Browser-Panels durch, keine selbstgebauten Skripte mit deinem apiKey. Fuer Browser nutzt App Check reCAPTCHA v3 – der hier eingetragene Wert ist der oeffentliche Site Key dieser reCAPTCHA-Site (beginnt mit 6L). Ohne diesen Key bleibt App Check im permissive-Mode – alles funktioniert, aber Bot-Traffic wird nicht geblockt. Wird typischerweise erst vor dem Production-Go-Live aktiviert.

So bekommst du den Site Key (3 Schritte – nur fuer Production)
  1. reCAPTCHA-Admin-Console oeffnen → „reCAPTCHA v3" waehlen → Domains hinzufuegen (127.0.0.1, localhost, deine Produktiv-Domain) → Site Key kopieren.
  2. Firebase-Console → App Check → Web-App auswaehlen → „reCAPTCHA v3" als Provider hinzufuegen → Site Key dort einfuegen.
  3. Denselben Site Key unten in dieses Feld einfuegen – die Browser-Panels nutzen ihn dann als Attestation-Provider.

📱 Werte, die NACH dem Setup im Admin-Panel konfiguriert werden (Play Integrity, Device Check, VAPID, reCAPTCHA, FCM)

Die folgenden Werte werden nicht in .env abgelegt, sondern im Firestore-Dokument operatorConfig/externalIntegrations – sichtbar im Admin-Panel unter Setup → Externe Integrationen .

  • Play Integrity Key (Pfad zu Secret Manager). Erforderlich, damit der Backend den von der Master-App gesendeten Play-Integrity-Token validieren kann (Schutz vor manipulierten Android-Installationen). Play Integrity in Cloud-Console aktivieren ↗
  • Apple DeviceCheck Key (Pfad zu Secret Manager). Optional, wird fuer das iOS-Pendant verwendet (App Attest / DeviceCheck).
  • FCM Server Key (Pfad zu Secret Manager). Wird vom Backend zum Senden von Push-Nachrichten an die Master-/Child-Apps gebraucht. Wenn Firebase Admin SDK ueber den Service Account laeuft (typisch), ist dieser Eintrag optional.
  • reCAPTCHA v3 Site Key (oeffentlich, kein Secret). Wird auch in das FIREBASE_APP_CHECK_SITE_KEY-Feld weiter oben eingetragen – beide Stellen muessen denselben Wert haben, falls Web-App- Check ueber reCAPTCHA aktiviert ist.

Diese Werte sind nicht zwingend fuer den ersten Start. Erst beim Production-Go-Live im Admin-Panel ergaenzen.

Vollqualifizierter Name des Cloud-Pub/Sub-Topics, das Real-Time Developer Notifications (RTDN) von Play empfaengt – z. B. Abo-Verlaengerungen, Kuendigungen, Reaktivierungen. Ohne Topic laeuft der Subscription-Webhook leer und Abo-Statusse bleiben veraltet. Format: projects/<gcp-projekt>/topics/<topic-name>.
